# WordPress Deploy Agent

## Tier 2 — Task Agent

Spawned by: `koray-orchestrator-agent` (Phase 6 — Deploy) or manually after audit completion

Uses: `wp-deploy` skill + `validate-schema` skill

## Pre-flight checklist (must verify before any deploy)

- [ ] All target pages have HCU score ≥ 80 (check `projects/<slug>/audit/`)
- [ ] `projects/<slug>/wp_credentials.json` exists and is readable
- [ ] WordPress version ≥ 6.0 (Application Passwords available)
- [ ] Target site is accessible (HTTP 200 on `wp_url`)
- [ ] Content payloads exist in `projects/<slug>/content/`
- [ ] Schema blocks generated by `schema-markup-creator` are present

## Deployment workflow

### Step 1 — Validate all schema (blocking gate)

For every page in the deploy batch:
```bash
python python-backend/schema_validator.py --schema_file projects/<slug>/audit/<page>-schema.json
```
- If ANY page has `valid: false` → stop, report errors, do not deploy anything
- Schema errors must be fixed before proceeding

### Step 2 — Build batch payload

Assemble `batch_payload.json` in `projects/<slug>/content/`:
- Order pages: hub/pillar pages first, then category pages, then spoke pages
- Set all pages to `"status": "draft"` (never publish directly)
- Include `internal_links` map from `internal-linking-matrix-builder` output

### Step 3 — Deploy batch

```bash
python python-backend/wp_deployer.py \
  --payload projects/<slug>/content/batch_payload.json \
  --credentials projects/<slug>/wp_credentials.json \
  --mode batch \
  --output projects/<slug>/sheets/deployed_pages.json
```

### Step 4 — Handle failures

For each `FAILED` result in the output:
- Read the `error` field from the JSON output
- Common errors and fixes:
  - `HTTP 401` → Application Password is incorrect or expired — ask user to regenerate
  - `HTTP 403` → User lacks Editor/Admin role — check WP user permissions
  - `HTTP 422` → Payload validation error — check for missing required fields
  - `HTTP 500` → Server error — retry once after 30s; if persistent, report to user

### Step 5 — Trigger browser QA

Pass all successful `url` values to `browser-qa-agent`:
```bash
python python-backend/browser_qa.py --url_list projects/<slug>/sheets/deployed_pages.json \
  --output projects/<slug>/audit/qa_report.json
```

### Step 6 — Publish pages that pass QA

For each page where QA score ≥ 80:
- Re-run `wp_deployer.py` in single mode with `"status": "publish"`
- Update `deployed_pages.csv` with live URL and publish timestamp

### Step 7 — Save deployment log

Write `projects/<slug>/sheets/deployed_pages.csv`:
```
title,slug,post_id,draft_url,live_url,qa_score,deployed_at,status
```

## Deployment order (critical)

```
1. Homepage / root page (if included)
2. Core/pillar pages (receive most internal links)
3. Category/section pages
4. Spoke/supporting pages (link up to core pages above them)
```

Never deploy spoke pages before the hub pages they link to — broken internal links damage crawlability.

## SEO plugin configuration

| Plugin | Fields set | How to confirm plugin is active |
| ------ | ---------- | ------------------------------- |
| Rank Math | `rank_math_title`, `rank_math_description`, `rank_math_focus_keyword` | Check WP Admin → Plugins |
| Yoast SEO | `_yoast_wpseo_title`, `_yoast_wpseo_metadesc` | Check WP Admin → Plugins |
| None | Custom `_schema_json_ld` field only | Use with theme that reads custom fields |

## Deliverables

- `projects/<slug>/sheets/deployed_pages.json` — raw API results
- `projects/<slug>/sheets/deployed_pages.csv` — human-readable log
- `projects/<slug>/audit/qa_report.json` — post-deploy QA results
- Deployment summary: `Total | Success | Failed | QA passed | Live`
